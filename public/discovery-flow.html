Discovery Flow — OIDC openid-configuration

分析 getDiscoveryField() 的三層快取策略與所有 caller 的 endpoint 消費關係， 涵蓋 Memory → LocalStorage → Network 的查找順序與版本化儲存 Key 設計。

01

Caller 全覽 — 誰消費哪個 OIDC Endpoint

getDiscoveryField(target) 是所有 OIDC endpoint 的統一入口。 5 個 helper 各自帶入不同的 keyof Discovery 欄位名稱， 取回 IdP 提供的實際 URL 後再發出對應的 HTTP 請求。

IdP
IdP 的 openid-configuration 包含所有 OIDC endpoint URL，是唯一的 source of truth
GDF
getDiscoveryField 是統一入口，所有 helper 不直接 fetch，而是透過它取得 endpoint

L2
localStorage 快取（可選），跨頁面重新整理後仍有效，版本號與套件版本綁定
02

getDiscoveryField() — 三層查找流程

查找優先順序：Memory → localStorage → Network。 任何一層命中即短路回傳，避免不必要的 HTTP 請求。 每次 fetch 成功後會同時 hydrate Memory 與（若啟用）localStorage， 讓後續查找直接命中 L1 而不需再往下查。

L1 Hit
最快路徑。Singleton 記憶體命中，同一個 browser session 中第一次 fetch 成功後即永久有效
L2 Hit
跨頁面重整有效。localStorage 命中後會同步 hydrate L1，使後續查找直接命中 Memory
Network
冷啟動或快取失效時觸發。使用 to() 工具函式包裝，避免 throw 中斷呼叫端流程
Error
所有失敗路徑最終收斂到 onCriticalError callback，由呼叫端決定錯誤處理策略

04

localStorage Key 設計 — 版本化自動失效

儲存 Key 包含套件版本號，確保套件升級後自動使用新 Key，舊版快取自然失效而不需主動清除。 clearDiscoveryInStorage() 則使用不含版本的前綴進行 startsWith 掃描，一次清除所有歷史版本的殘留快取。

情境 Key 格式 範例 說明
讀 / 寫快取 {opIssuer}_{authEnv}_discovery_{version} MVB_dev_discovery_0.1.3 精確版本，套件升級後自動改用新 key，舊快取被棄用但不佔用

快取失效條件 套件版本號變更 0.1.2 → 0.1.3 不需手動失效；只需升級套件版本，舊快取的 key 不再被讀取
設計亮點
版本綁定 Key 是一種 implicit cache invalidation，無需額外的版本比對邏輯，依賴 key 不同即自然失效
清除策略
clearDiscoveryInStorage 使用 Object.keys(localStorage) 前綴過濾，跨版本清理不留殘留
